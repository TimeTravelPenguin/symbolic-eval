# Symbolic Evaluation for Typst

This package was made possibly by the incredible library, Symbolica.

## Why does this package exist?

Computational performance is paramount for certain kinds of tasks. It is my belief that
software should be accessible to everyone, but especially to those employed with education
and research.

Much of the time, certain tasks require preparing document assets in advance by
programming and saving the result. Typst is in a unique position where it can bridge that
gap completely (or at least partially), via packages that offer domain-specific
functionality.

Consider the example where a user wanted to create an image to visualise the
complex-valued function `f(z)`. For those of you who are of non-mathematical background,
this function `f(z)` can be thought of as a function that takes a point `(x, y)` to a new
point `(u, v)`. Both input and output are 2-dimensional, meaning we need four dimensions
to visualise `f(z)` using traditional techniques. One way around this is to use colour.
The pixel corresponding to the point `(x, y)` can be coloured using a colouring function
that takes `u` and `v` as inputs. One such output could be the following:

![phase plot of `(z^2 - 1) / (z^2 + 1)`](./assets/phase_portrait.png) 

This particular render contains many (too many) additional visual elements to show
different features of the function (one or two on their own would normally be fine, but
this is a demonstration). The image is downscaled massively, but was originally rendered
at a high-resolution with 4x subsampling. For a _document_, these details can be lowered
dramatically, but the point is that this is not a simple pipeline:
1. Each pixel of the image must be coloured.
2. Additional visual features may require additional computation complexity. In this case,
   finite-differences was required.
3. Implementing sub-sampling for higher-quality renders.

These few features can be too much for a language such as Typst. However, through WASM
plugins, this can be handled with ease!

Thus, a package such as this allows for, and demonstrates how, other packages to perform
computations that would otherwise be too slow to compute normally.

## Why Symbolica? What is it?

Symbolica is a library written in Rust, for both Rust and Python, that provides
high-performance computor algebra. Typically, it is a library used by very smart people to
do very smart things that are very difficult to do normally. For something like an image
renderer, we can rely on approximations of numerical methods that converge to values
accurate enough to be reasonable. However, in some cases, we don't want this. We instead
want a nice formula or some other result that is an extension of some input.

Normally, Typst would be an odd place for something like Symbolica. However, Typst
currently lacks a way for WASM plugin developers to communicate _functions_ (or really,
any kind of computation). If I wanted to ask a plugin to evaluate `x + 1` for me with the
input of `x = 2`, there is no easy way to do this.

Now, imagine if a plugin developer wanted to make a phase-portrait renderer (the
complex-valued function shown in the previous section). They would have to use an
_expression evaluator_ to parse a string input into something that can be used to do
calculations. This is _HARD_ to do efficiently without even mentioning that we need the
expression evaluator to handle _complex-valued functions_.

Symbolica is capable of---and designed for---optimising and computing this task.

## The state of this package

This package is still in development. Currently, as of writing this, it is still tricky to
get Symbolica to compile, because it requires RNG, which is something unavailable for WASM
plugins (this is an oversimplification, but the point remains).

I am not 100% sure what a perfect solution is, but it is something I am looking into. If
you have experience with Rust's `getrandom` crate and compiling to WASM, I very much
welcome you to open an issue to start a discussion.

<details>

<summary>
    For now, I am using a custom backend, as is implemented in
    [lib.rs](./rust/src/lib.rs).
</summary>

Compile time flags are additionally found in [justfile](./justfile)'s private `_build`
recipe.

The backend is then implemented as the following, which was generated by AI because I have
very little experience with custom backends.

```rs
#[cfg(target_arch = "wasm32")]
use std::sync::atomic::{AtomicU64, Ordering};

#[cfg(target_arch = "wasm32")]
#[unsafe(no_mangle)]
unsafe extern "Rust" fn __getrandom_v03_custom(
    dest: *mut u8,
    len: usize,
) -> std::result::Result<(), getrandom::Error> {
    static STATE: AtomicU64 = AtomicU64::new(0x4d59_5df4_d0f3_3173);

    let mut state = STATE
        .fetch_add(0x9e37_79b9_7f4a_7c15, Ordering::Relaxed)
        .wrapping_add(len as u64);
    let bytes = unsafe { std::slice::from_raw_parts_mut(dest, len) };

    for byte in bytes {
        state ^= state >> 12;
        state ^= state << 25;
        state ^= state >> 27;
        *byte = state.wrapping_mul(0x2545_f491_4f6c_dd1d) as u8;
    }

    Ok(())
}
```

</details>

For now, this package does almost nothing, with the exception of basic expression
evaluation, as I am trying to understand the issues surrounding compilation.

## Building this package

If you want to build this package for yourself, you will need a few dependencies:
- The [Rust](https://rust-lang.org) toolchain,
- [Just](https://github.com/casey/just), required to run the build scripts,
- [Dasel](https://github.com/TomWright/dasel), used for querying toml data,
- (Optional) [Oxipng](https://github.com/oxipng/oxipng), used to optimise png files with
  non-lossy compression. This is currently unused, but will likely be in the future.

Building using `just build` will compile the wasm package into the `./lib/wasm` directory.
